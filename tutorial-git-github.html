Git & GitHub paso a paso

Una guía práctica para llevar tu código del ordenador a la nube, trabajar con ramas y colaborar sin pisarte con nadie.

Tutorial guiado · Nivel intermedio

i Git no es GitHub

Se nombran juntos todo el rato, pero son cosas distintas.

Git es el programa que se instala en tu ordenador y guarda el historial de tu proyecto: cada cambio, quién lo hizo y cuándo. Funciona sin internet.

GitHub es una página web donde subes una copia de ese historial para guardarlo en la nube, compartirlo y trabajar en equipo. Hay alternativas (GitLab, Bitbucket), pero el flujo es el mismo.

💡 La analogía: Git es como el control de versiones de un documento (guardar partidas). GitHub es como Google Drive donde dejas ese documento para que otros lo vean y lo editen contigo.

↻ El flujo que vamos a repetir siempre

Tres "zonas" en tu equipo y una en la nube. Todo Git se reduce a mover cambios entre ellas.

Lee el dibujo de izquierda a derecha: editas archivos (Working Dir), eliges cuáles guardar con add (Staging), los confirmas con commit (Repo local) y los envías a la nube con push (GitHub). La flecha de vuelta (pull) trae lo que otros hayan subido.

1 Configurar Git (solo la primera vez)

Git necesita saber tu nombre y correo para firmar cada commit.

Comprueba que lo tienes instalado:

git --version

Configura tu identidad (usa el mismo correo que en tu cuenta de GitHub):

git config --global user.name "Tu Nombre"
git config --global user.email "tucorreo@ejemplo.com"

Define main como nombre de la rama principal por defecto:

git config --global init.defaultBranch main

💡 --global aplica la configuración a todos tus proyectos. Para revisar cómo está todo: git config --list.

2 Crear o clonar un repositorio

Un repositorio (repo) es una carpeta que Git vigila. Hay dos formas de empezar.

Opción A — Convertir una carpeta tuya en repo

# dentro de la carpeta de tu proyecto
git init

Esto crea una subcarpeta oculta .git donde se guarda todo el historial. Si la borras, pierdes el historial.

Opción B — Clonar un repo que ya existe en GitHub

git clone https://github.com/usuario/proyecto.git

Descarga una copia completa (con todo su historial) en una carpeta nueva. Es lo que harás para bajar el proyecto de la clase o el de un compañero.

3 El ciclo de trabajo: add y commit

Este es el corazón de Git y lo repetirás decenas de veces al día.

Antes de nada, mira siempre en qué estado estás. git status es tu mejor amigo:

git status

Prepara los archivos que quieres guardar (pasarlos a Staging):

# un archivo concreto
git add index.html

# todos los cambios de golpe
git add .

Confirma el cambio con un mensaje claro que explique qué hiciste:

git commit -m "Añade formulario de contacto"

Y consulta el historial de commits cuando quieras:

# historial resumido, una línea por commit
git log --oneline

⚠️ Error típico: hacer commit esperando que suba a GitHub. No. El commit solo guarda en tu equipo. Para subirlo hace falta push (paso 4).

💡 Buen mensaje de commit: en imperativo y concreto — "Corrige cálculo del IVA", no "cambios" ni "asdf". Tu yo del futuro te lo agradecerá.

4 Conectar y subir a GitHub

Now enviamos el historial local a la nube.

Primero crea un repositorio vacío en GitHub (botón New → nombre → Create repository). No marques "Add README" si vas a subir una carpeta que ya tiene commits, para evitar conflictos.

Conecta tu repo local con el de GitHub. A esa conexión se le llama origin por convención:

git remote add origin https://github.com/usuario/proyecto.git

Sube tu rama main por primera vez:

git push -u origin main

La opción -u deja recordada la conexión. A partir de aquí, subir cambios es tan simple como:

git push

💡 GitHub ya no acepta contraseña al hacer push. Usa un Personal Access Token (Settings → Developer settings → Tokens) como contraseña, o configura una clave SSH. Una vez hecho, no vuelves a loguearte.

5 Ramas: trabajar sin romper lo que funciona

Una rama es una línea de trabajo paralela. Pruebas algo nuevo sin tocar main.

Crea una rama y muévete a ella en un solo comando:

git switch -c nueva-funcionalidad

Lista tus ramas (la actual sale marcada con *) y cámbiate entre ellas:

git branch              # ver ramas
git switch main         # volver a main

Cuando tu trabajo esté listo, vuelve a main y fusiónala (merge):

git switch main
git merge nueva-funcionalidad

💡 Regla de oro: main siempre debe contener código que funciona. Lo experimental va en ramas. En equipos reales casi nunca se hace merge a mano: se hace con un Pull Request (paso 7).

6 Trabajar en equipo: pull y conflictos

Si otros suben cambios, tu copia se queda desactualizada. Hay que sincronizar.

Antes de empezar a trabajar cada día, trae lo último de GitHub:

git pull

A veces dos personas editan la misma línea del mismo archivo. Git no sabe cuál vale y avisa de un conflicto. Lo verás así dentro del archivo:

<<<<<<< HEAD
tu versión de la línea
=======
la versión de tu compañero
>>>>>>> main

Para resolverlo: edita el archivo, deja solo el texto correcto, borra las líneas marcadoras (<<<, ===, >>>) y haz un commit normal:

git add .
git commit -m "Resuelve conflicto en estilos"

⚠️ La mejor forma de evitar conflictos es hacer git pull a menudo y commits pequeños. Cuanto más tiempo pasa sin sincronizar, más feo el conflicto.

7 Pull Requests: la forma profesional de fusionar

Un Pull Request (PR) es pedir "por favor, mete mi rama en main" para que alguien lo revise antes.

El flujo completo en equipo es:

  1. Creas una rama y haces tus commits (pasos 3 y 5).
  2. Subes esa rama a GitHub: git push -u origin nueva-funcionalidad
  3. En la web de GitHub aparece un botón Compare & pull request. Púlsalo.
  4. Escribes un título y una descripción de lo que cambiaste.
  5. Un compañero (o el profe) revisa, comenta y aprueba.
  6. Se pulsa Merge pull request y tu código entra en main.

💡 La gracia del PR es la revisión: alguien mira tu código antes de que entre en la rama principal. Así se detectan errores y se aprende del código de los demás. Es como funciona casi todo el software del mundo.

8 .gitignore: lo que NO debe subir

Hay archivos que no quieres en el repo: contraseñas, dependencias pesadas, archivos temporales.

Crea un archivo llamado .gitignore en la raíz del proyecto y escribe dentro lo que Git debe ignorar:

# dependencias
node_modules/

# archivos de compilación
/bin/
/obj/
*.exe

# secretos y configuración local
.env
appsettings.local.json

# basura del sistema
.DS_Store
Thumbs.db

⚠️ Nunca subas contraseñas ni claves API. Una vez en el historial, quedan ahí aunque borres el archivo después. Añádelas a .gitignore desde el primer commit.

💡 GitHub ofrece plantillas de .gitignore ya hechas por lenguaje (Node, Python, Visual Studio…) al crear el repo. Aprovéchalas.

Los que usarás el 95% del tiempo. Guárdala a mano.

Comando Qué hace

git add .Preparar todos los cambios para el commit
git commit -m "msg"Guardar los cambios preparados con un mensaje
git pushSubir tus commits a GitHub
git pullBajar los cambios de GitHub a tu equipo
